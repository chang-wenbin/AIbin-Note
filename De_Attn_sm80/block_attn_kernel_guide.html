1 整体架构：两阶段流水线

这个 kernel 实现了 Multi-Block Paged GQA Attention。当序列长度很长时，单个 thread block 无法处理完所有 KV token， 因此将序列切分为多个 partition，每个 partition 由一个独立的 thread block 并行计算，最后通过一个 reduce kernel 合并所有 partition 的结果。

图 1：两阶段 Multi-Block Attention 架构总览
为什么需要两个阶段？ 每个 partition 独立计算时，只能得到局部的 softmax 结果（因为没有看到其他 partition 的 score）。 reduce kernel 通过 online softmax 的数学性质，利用各 partition 导出的 max 和 sum 将局部结果修正为全局正确的 attention output。

3 Shared Memory 布局与 Swizzle

SmemLayoutAtomQ：基础 Atom

所有 shared memory layout 都基于一个 8×64 的 atom，叠加了 Swizzle<3,3,3> 来消除 bank conflict： 

using SmemLayoutAtomQ = composition(
    Swizzle<3, 3, 3>{},
    Layout<Shape<Int<8>, Int<64>>,
           Stride<Int<64>, _1>>{});
Swizzle<3,3,3> 的含义： 对地址做 bit 级变换：addr[5:3] ^= addr[8:6]。这使得相邻行的同一列映射到不同的 bank， 128 个线程同时读取时不产生 bank conflict。

tile_to_shape：平铺 Atom 到目标大小

using SmemLayoutKV = tile_to_shape(
    SmemLayoutAtomQ{},                         // 8×64 的 atom
    Shape<Int<kBlockSize>, Int<kHeadDim>>{});  // 目标: 64×128
图 2：Shared Memory Layout 通过 tile_to_shape 生成

Shared Memory 分区

smem_ ─┬─ sQ        [kBlockM × kHeadDim]   = [16 × 128]  // Q 矩阵
       ├─ sQbias    [kBlockM × kHeadDim]   = [16 × 128]  // Q bias
       ├─ sQK       [kBlockM × kBlockSize] = [16 × 64]   // P = softmax(S)，复用
       ├─ sK        [kBlockSize × kHeadDimKV]             // K cache block
       └─ sV / sVt  [kBlockSize × kHeadDimKV]             // V cache block (+ 转置视图)

6 GEMM 实现：Software Pipelining

核心 GEMM 函数沿 K 维度切片，实现了 depth-1 的 smem → register 预取流水线。 先发射 copy（走 Load/Store 单元），再发射 gemm（走 Tensor Core），两者可以并行执行。

图 5：GEMM 中 smem→reg 预取流水线时序

retile_D 的作用

smem_thr_copy_A.retile_D(tCrA) 把同一块寄存器从 MMA 计算的 layout 重新解释为 smem copy 指令（如 ldmatrix）的 layout。不搬数据，只换视图。 后续 copy(smem_thr_copy_A, tCsA(_, _, i), tCrA_copy_view(_, _, i)) 按 copy-friendly 的 layout 把数据搬进寄存器， 而 cute::gemm(tiled_mma, tCrA(_, _, i), tCrB(_, _, i), acc) 按 MMA-friendly 的 layout 消费同一组寄存器。

7 MMA Layout 转换：convert_layout_acc_rowcol 

auto l = logical_divide(acc_layout, Shape<_2>{});   // (4, MMA_M, MMA_N) → ((2,2), MMA_M, MMA_N)
return make_layout(
    make_layout(get<0,1>(l), get<1>(l)),   // 行: (2_row, MMA_M)
    make_layout(get<0,0>(l), get<2>(l))    // 列: (2_col, MMA_N)
);
图 6：MMA 累加器 3 阶 layout → 2 阶 (row, col) 行列 layout

转换后，外层循环 mi 遍历行，内层循环 ni 遍历列，方便逐行做 softmax 的 reduce_max、reduce_sum、apply_mask 等操作。

8 边界 Mask 处理 

if (partition_idx == partition_num - 1 && cur_remain_seq_len < kBlockSize) {
    block_attn::apply_mask<kMiLen>(scores, warp_id, col, cur_remain_seq_len);
}

只有最后一个 partition 的最后一个 block 才可能不满 64 个 token。 apply_mask 还原每个线程持有的 score 元素对应的真实列索引：

列索引还原公式
col_index = warp_id × 8 + ni × 32 + col × 2
分量含义范围
warp_id × 8每个 warp 负责 8 列的一段0, 8, 16, 24
ni × 32不同 MMA tile 之间间隔 320, 32
col × 2lane 在 8 列 MMA tile 内的偏移0, 2, 4, 6

如果 col_index ≥ remain_seq_len，将对应的 scores(mi, ni*2) 和 scores(mi, ni*2+1) 置为 -INF。 经过 softmax 后 e-∞ = 0，这些无效位置对输出没有贡献。

9 Online Softmax 与 Output Rescale

这是 FlashAttention 的核心算法——在不知道完整序列的情况下，增量更新 softmax。

图 7：Online Softmax 增量更新过程
scale_apply_exp2 中的关键行：
tensor(mi, ni) = expf(tensor(mi, ni) * scale - max_scaled)
等价于 exp((score - max) × scale)，减去 max 保证指数参数 ≤ 0，结果在 (0, 1] 范围，防止浮点溢出。 max_scaled = max[mi] * scale 提到循环外避免重复计算。

11 Partition 收尾：写出 max / sum / O

每个 partition kernel 结束时，将三组关键数据写入 global memory，供 reduce kernel 使用：

输出数据归约方式
params.maxs[p] mp = scores_max × inv_sqrt_dh scores_warp 跨 warp MaxOp 归约
params.sums[p] dp = Σ softmax_sum warp 内 Allreduce<4> + 跨 warp SumOp 归约
params.partition_attn_out[p] Op (未归一化) acc_o → FP16 → sQ (smem) → global memory
为什么 acc_o 写出时复用 sQ？ 此时 sQ 中的 Q 数据已经不需要了（所有 GEMM 都算完了），正好可以复用这块 shared memory 做 register → smem → gmem 的中转，节省 smem 用量。

13 Warp-Level Allreduce

Kernel 中有两种归约机制，分别适用于不同规模。

Allreduce<THREADS>：Warp 内 Shuffle 归约

基于 __shfl_xor_sync，递归减半，不需要 shared memory。用于 warp 内 4 个线程的 scores_sum 归约。

图 10：Warp 内 Shuffle Allreduce（__shfl_xor_sync）

BlockAllReduce：Block 级 CUB 归约

基于 CUB 库的 BlockReduce，需要 shared memory。 CUB 的 Reduce 只保证 thread 0 拿到结果，所以额外通过一个 result_broadcast 变量 + __syncthreads() 广播给所有线程。 用于 reduce kernel 中跨所有 128 线程求全局 max / sum。 

T result = BlockReduce(temp_storage).Reduce(val, ReductionOp());
if (threadIdx.x == 0) { result_broadcast = result; }
__syncthreads();
return result_broadcast;  // 所有线程都拿到结果
